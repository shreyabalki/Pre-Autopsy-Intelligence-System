Problem

Millions of deaths go uninvestigated

Investigators in low-resource settings have no fast, unified tool to perform a structured first-pass across body evidence, scan images, and toxicology simultaneously.

🔬

Forensic Expert Shortage

Approximately 150 forensic pathologists per 100 million people globally. Access is concentrated in urban centres. Rural and low-income regions operate with little to no specialist coverage.

⏳

Delayed Case Review

Waiting days or weeks for specialist analysis allows evidence to degrade and critical investigative windows to close. There is no accessible mechanism for fast first-pass triage.

🧩

Fragmented Tooling

Medical imaging, toxicology analysis, and external trauma review are handled by entirely separate tools with no shared workflow or unified reporting format for investigators.

🌍

Low-Resource Access Gap

Existing medical AI tools are designed for clinical environments in well-resourced settings. They are not built for investigators who need lightweight, accessible forensic triage.

📋

Inconsistent Reporting

Field investigator reports are often informal, unstructured, and inconsistently communicated — reducing their value for downstream specialist review and case escalation decisions.

📊

No Prioritisation System

Without structured severity ratings, investigators have no systematic basis for deciding which cases require immediate specialist escalation versus routine follow-up.

Solution

Structured forensic screening — any investigator, anywhere

A single tool that accepts multiple forensic evidence types and returns a consistent, structured report in under 60 seconds. Designed to surface observable signals and recommend escalation — not to replace specialist judgement.

✦
Multi-modal input
Accepts medical images (JPEG/PNG) and unstructured text. Six independent modules with tailored analysis per evidence type.
✦
Structured JSON output
Every analysis returns a consistent 14-field report: findings, severity, confidence, region, interpretation, next steps, and investigator action.
✦
Three-level severity classification
Normal, Suspicious, or Critical — providing a clear basis for case prioritisation and escalation decisions.
✦
Screening only — not diagnosis
Extracts observable signals from the input. Does not determine cause of death, provide clinical diagnoses, or generate legal conclusions.
✦
AI-annotated images
Image outputs include approximate bounding-box overlays on regions of interest. Marked as AI-estimated and for triage reference only.
✦
Expert review enforced
Every report includes a built-in expert-review disclaimer and escalation guidance. The system cannot be used to generate final conclusions.

Pipeline

How it works

The full process from investigator input to structured report in under 60 seconds.

1

Upload Input

Investigator selects the forensic module and uploads an image or pastes text. Each module accepts only the input types it is designed to process.

2

Preprocess

Images are resized to 512px max, compressed to JPEG 80%, and base64-encoded before API submission. Text inputs are passed directly as plain content.

3

Analyze with Gemini Flash

Input is sent to Google Gemini 2.5 Flash with a module-specific system prompt. The model is instructed to return strict JSON only, with no explanatory text.

4

Generate JSON Report

The API response is parsed and validated against a 14-field schema. If parsing fails, a graceful fallback result is returned without crashing the application.

5

Render Findings

The structured report is rendered as a severity-coded visual card with findings, interpretation, differential considerations, and recommended next steps.

6

Review and Escalate

The investigator reviews the report and decides whether to escalate to a forensic specialist, based on severity and investigator action guidance.

Report schema (JSON)
"module": "Chest X-ray",
"case_summary": "Bilateral opacity with cardiomegaly present.",
"anomalies": "Yes",  "severity": "Suspicious",  "confidence": "Medium",
"suspected_region": "Bilateral lung fields",
"key_findings": ["Bilateral lower lobe opacity", "Cardiothoracic ratio > 0.5"],
"recommended_next_steps": ["Correlate with toxicology", "Refer to forensic pathologist"],
"investigator_action": "Escalate to forensic pathologist. Priority: High.",
"disclaimer": "AI screening only. Expert review required."
Architecture

How it was built

Built with Python and Streamlit. No custom model training. No GPU infrastructure. Lightweight enough to run from a laptop or deploy on Streamlit Cloud at zero cost.

🐍
Python 3.10+
Core application language. Minimal dependencies: requests, Pillow, python-dotenv, Streamlit.
🖥
Streamlit
Python-native web framework. Handles UI, session state, file upload, sidebar history, and responsive layout.
⚡
Gemini 2.5 Flash
Google's multimodal AI accessed via API. Handles image vision and text NLP in a single inference call.
🖼
Pillow (PIL)
Image resizing and compression before API upload. Bounding-box annotation drawing without OpenCV dependency.

Model Strategy — No Custom Training

This system does not train any custom deep learning model. There are no model weights, no training pipelines, and no GPU requirements. All inference runs through the Google Gemini API.

Outputs are shaped by module-specific system prompts, a strict JSON schema, hallucination-reduction rules, and mandatory expert-review disclaimers embedded in every prompt.

Why API-first:
→Builds and iterates in days without retraining
→No GPU infrastructure or model hosting costs
→Deployable on free-tier cloud platforms
→Each module is independent and updatable via prompt
→Accessible in low-resource environments
Modules

6 forensic modules

Each module operates independently with its own system prompt, accepted input types, and report schema. Switching modules resets all application state.

🫁Chest X-ray

Screens for lung opacity, pleural fluid, pneumothorax, rib fracture patterns, and mediastinal widening.

JPEG / PNG NIH ChestX-ray14
🧠Brain MRI / CT

Screens for masses, hemorrhage, edema, hemispheric asymmetry, midline shift indicators, and density irregularities.

JPEG / PNG BraTS 2023
🦴Full Body CT

Screens for fractures, internal bleeding indicators, organ density anomalies, and soft tissue disruption.

JPEG / PNG RSNA Hemorrhage
🧪Toxicology Report

Screens for substances above reference thresholds, alcohol markers, opioids, benzodiazepines, overdose risk, and poly-substance interactions.

Text / .txt MIMIC-IV
📷External Trauma

Screens for visible injury patterns, bruising distribution, lacerations, burns, and ligature-like marks using INTERPOL DVI protocol language.

JPEG / PNG INTERPOL DVI
⚡Deep Brain Screening

Screens for structural asymmetry, deep-region irregularities, white matter signal changes, and occipital and parietal density anomalies.

MRI / CT image BraTS 2023

Privacy

Data collection & privacy

The system processes only what the investigator explicitly provides. Nothing is collected, tracked, or retained beyond active processing.

🔒
User-provided input only
The system processes only what the investigator explicitly uploads or enters. No automatic or passive data acquisition.
📵
No background monitoring
No scraping, no session recording, no behavioral analytics, and no background data collection at the application layer.
🗑
No long-term storage
Uploaded images and case content are not retained after the session ends. No cloud database. No file persistence beyond the local session.
👤
No personal data stored
No patient identifiers, biometric data, or case metadata is stored by the application itself.
📤
Third-party inference
Inputs are sent to Google Gemini API. Review Google's API data handling policies for applicable protections and terms.
⚖️
User responsibility
The operator is responsible for ensuring that data submitted to the system is used lawfully and with appropriate authorisation.

Limitations

Known limitations

These constraints are documented clearly and are not treated as edge cases.

⚠
Not diagnostic. The system performs screening-level pattern recognition. It does not provide clinical diagnoses or determine cause of death.
⚠
Not legally authoritative. Outputs carry no legal weight and cannot be submitted as legal evidence in any jurisdiction.
⚠
Input quality dependency. Low-resolution or poorly captured images directly reduce the reliability and confidence of findings.
⚠
AI may miss findings. The model may fail to detect subtle anomalies that require specialist clinical expertise to identify.
⚠
Toxicology completeness. Analysis quality depends entirely on the completeness and formatting of the submitted lab report.
⚠
Expert validation mandatory. All outputs are screening-level. No report should influence a decision without qualified forensic specialist review.
Ethics & Safety

Ethics & safety framework

These constraints are enforced at the system prompt level for every module and every inference call. They are not advisory — they are architectural.

Prompt-level constraints
→AI prohibited from naming specific diseases or syndromes
→AI prohibited from stating or implying cause of death
→AI prohibited from inventing patient demographics or identifiers
→AI must describe only what is visually or textually observable
→Expert-review disclaimer required in every single JSON response
Output integrity rules
→No automated final decisions — human expert must review all outputs
→No legal conclusions of any kind
→No medical diagnoses
→All language hedged: "consistent with", "suggests", "warrants review"
→Graceful fallback on API failure — no silent errors
This system does not
✗Determine cause of death
✗Replace forensic pathologists
✗Generate legal evidence
✗Provide medical diagnosis
✗Guarantee accuracy of findings
✗Support automated final decisions
⚠️

AI screening only. Pre-Autopsy Intelligence System requires expert validation and must not be used as a standalone medical, legal, or cause-of-death decision system.
